# 主流厂商对是否保留shared expert的不同意见
在混合专家模型（MoE）架构中，关于是否引入 共享专家（Shared Expert），主流厂商和研究机构之间确实存在显著的技术路线分歧。这一分歧的核心在于：究竟是让所有参数都参与竞争与分化，还是人为保留一部分“常驻参数”来提取通用知识。
以下是主流厂商和模型代表对此的不同观点与实践：
## 1. 坚持保留 Shared Expert 的阵营
以 DeepSeek (深度求索) 为代表。他们在 [DeepSeekMoE](https://zhuanlan.zhihu.com/p/1927154457638072839) 架构中明确引入了 Shared Experts。

* 核心逻辑：
* 通用知识冗余问题：如果所有专家都是被动态路由激活的（Routed Experts），不同的专家可能会重复学习到一些“常识性”或“底层语法”知识，导致计算资源浪费。
   * 知识隔离与聚合：Shared Expert 始终处于激活状态，专门负责捕捉跨任务的公共知识，而 Routed Experts 则可以更纯粹地去学习垂直领域的专业知识。
   * 参数效率：DeepSeek 认为这种设计能显著提升参数效率。例如 [DeepSeek-V3](https://cloud.tencent.com/developer/article/2483443) 仅用了极少的训练成本就达到了领先性能，Shared Expert 稳定了底层语义表达，降低了训练难度。 [1, 2, 3] 

## 2. 倾向于纯粹 Routed Expert 的阵营
以 Google (Switch Transformer) 和 Mistral (Mixtral 8x7B) 为代表。

* 核心逻辑：
* 天然分化理论：这些厂商倾向于保持 MoE 的纯粹性。他们认为只要数据量足够、路由机制合理，专家们会通过梯度竞争自然地分化出负责通用知识和负责专业知识的模块，不需要人为硬性规定一部分专家为“共享”。
   * 扩展性（Scalability）：纯 Routed 架构更利于分布式扩展。在万亿级参数模型中，所有参数根据 Token 动态加载，避免了 Shared Expert 成为推理时的单点计算瓶颈，因为 Shared Expert 意味着每个 Token 都必须经过这些参数。 [4] 

## 3. 争议与技术权衡点
目前业内对是否保留 Shared Expert 的讨论主要集中在以下三个方面：

* 推理延迟与计算量：
* 支持者认为：Shared Expert 虽然增加了每个 Token 的固定计算量，但因为它捕捉了公共知识，可以缩减 Routed Experts 的数量，从而在总激活参数不变的情况下获得更好的性能。
   * 反对者认为：在大规模集群推理时，Shared Expert 的存在使得每个节点都必须常驻这部分权重，且无法通过专家并行（Expert Parallelism）完全抵消其带来的固定时延。
* 训练稳定性：
* [DeepSeek](https://www.zhihu.com/question/1978716390480446480/answer/1979321695908156822) 的研究指出，如果没有 Shared Expert，高频词（如符号、助词）会被分散到不同的 Routed Expert 中处理，容易导致梯度震荡；而 Shared Expert 能作为“压舱石”稳定这些基础 Token 的表征。
* 系统指令遵循（System Prompt）：
* 一些分析指出，像 System Prompt 这样需要全局生效的长文本指令，更依赖始终在线的 Shared Expert 来维持一致的上下文理解。 [1] 

总结来看：以 DeepSeek 为首的一派通过 Shared Expert 追求极致的参数利用率和性价比；而传统的 Google/OpenAI 路线（推测 GPT-4 亦为 MoE 但未公开细节）则更偏向于通过海量数据和动态分化来让模型自主形成组织架构。



# 在哪些具体任务上共享专家表现出优势？

在混合专家模型（MoE）中，共享专家（Shared Expert） 并非简单地叠加参数，而是通过“隔离”公共知识来优化路由专家的分工。根据 [DeepSeekMoE](https://arxiv.org/pdf/2401.06066) 的研究和实际测试，共享专家在以下任务和场景中表现出显著优势：
## 1. 通用常识与底层语义任务

* 日常查询与基础语义表达：共享专家始终处于激活状态，专门捕捉不同上下文中的公共知识（Common Knowledge）。
* 语法与结构处理：在处理高频词（如助词、介词、标点符号）或基础句式时，共享专家能提供稳定的语义支撑，避免多个路由专家为了学习这些“无差别”信息而产生参数冗余。 [1, 2, 3, 4] 

## 2. 多语言翻译任务

* 通用语言特征提取：在多语言场景下，不同的语言对往往共有某些抽象特征。共享专家可以作为通用特征提取器，由所有任务共享，而路由专家则专注于特定语系的差异化处理，从而提高翻译的准确度和跨语言的一致性。 [5] 

## 3. 需要“全局一致性”的复杂任务

* 系统指令遵循（System Prompt）：由于 Shared Expert 处理每一个输入的 Token，它非常适合承载需要全局生效的指令（如“请用专业严谨的口吻回答”）。
* 基础知识基石：[DeepSeek-V3](https://developer.aliyun.com/article/1650760) 的设计显示，共享专家确保了模型拥有可靠的知识基础。在涉及多领域交叉的问题时，共享专家负责底层逻辑和常识，让路由专家专注于专业领域的深度推演。 [3] 

## 4. 极端参数效率场景

* 低计算预算的高性能任务：在 [DeepSeek-V2](https://arxiv.org/html/2405.04434v3) 的评估中，拥有共享专家的 DeepSeekMoE 2B 模型在涵盖 12 个零样本/少样本基准测试（包括文本理解、逻辑推理等）中，性能超越了规模更大的 GShard 2.9B。
* 这说明在资源受限的情况下，共享专家通过减轻参数冗余，让有限的参数发挥出了更大的效能。 [1, 2, 6] 

## 共享专家的核心价值总结

| 维度 [4, 7, 8, 9] | 优势表现 |
|---|---|
| 知识效率 | 减少路由专家间的冗余，将公共信息“压缩”至共享专家 |
| 训练稳定性 | 作为“压舱石”，防止路由专家因频繁切换而导致的梯度震荡 |
| 模型扩展 | 允许更细粒度的专家分割，提升了模型对复杂知识的捕获精度 |
