# MoE Forward

## minimind 中的 MoE Forward 
minimind-3-moe 是一个约 198M 总参数 / 64M 激活参数 的小型 MoE，并且其结构“aligned with Qwen3 / Qwen3-MoE”。

## 为什么需要MoE Forward
普通稠密模型里，每个 token 都要经过同一套 FFN/MLP 参数，所以：

- 想让模型更强，通常得把每层做宽、做深；
- 但这样每个 token 都要付出同样的计算代价；
- 一旦参数和计算都一起涨，训练和推理成本会很快爆掉。

MoE 的思路不一样：不是让所有参数每次都参与计算，而是让每个 token 只激活少数几个专家。这样就同时得到两件事：

- 总参数量可以很大
- 因为你可以放很多专家，每个专家都学不同模式。
- 单次计算量仍然比较低
- 因为每个 token 只走 top-k 个专家，而不是全部专家。

可以把它理解成一种“专家分工”：
- 有的专家擅长代码；
- 有的专家擅长数学；
- 有的专家擅长自然语言；
- 路由器根据当前 token 的特征，决定该找谁帮忙。

代价：
- 路由会变复杂；
- 训练时容易出现“某些专家太忙，某些专家闲着”的不平衡；
- 通信和工程实现更麻烦；
- 推理部署也更复杂。


## MoE 实现逻辑
1. 设置门控权重，从而选择每个 token 要走的 top-k 个专家、以及每个专家的权重。(如果有共享专家，需要将 token 配发给共享专家)
2. 各个专家进行FFN计算，得到输出。
3. 对专家输出进行加权，得到最终输出。(如果有共享专家，需要加上共享专家输出结果)
4. 计算 aux_loss ，用于平衡不同专家的计算量。

### 专家网络和FFN网络参数量对比
- Dense FFN：总参数 = 激活参数
- MoE：总参数 ≫ 激活参数
- 单个专家本体：和一个 FFN 的矩阵规模本质一样，都是 3dm 级别
- 差别只在“谁被激活、激活多少个”
